Foundations

What is agentic coding?

A single prompt is a request. An agent is a loop. It plans, acts, observes the result, and decides what to do next — autonomously, using tools and memory.

Vibe coding treats every AI interaction as a one-shot prompt. You ask, it answers, you move on. There's no loop, no verification, no accountability. This breaks at scale.
ApproachHow it worksFailure mode
PromptAsk once, accept the answerNo verification — bugs ship
SkillStructured procedure, consistent outputNo loop — one task at a time
AgentPlan → act → verify → loopNeeds harness or it drifts
Skill + AgentAgent runs reliable sub-proceduresCorrect approach ✓

agent_loop.py — minimal pattern
# Goal → Plan → Act → Observe → Verify → Loop import anthropic client = anthropic.Anthropic() def run_agent(goal: str, tools: list, max_steps: int = 10): messages = [{"role": "user", "content": goal}] for step in range(max_steps): response = client.messages.create( model="claude-sonnet-4-6", max_tokens=4096, tools=tools, messages=messages ) # Agent is done — no more tool calls if response.stop_reason == "end_turn": return response.content[0].text # Execute tool calls and feed results back tool_results = execute_tools(response.content) messages.append({"role": "assistant", "content": response.content}) messages.append({"role": "user", "content": tool_results}) raise RuntimeError(f"Agent exceeded {max_steps} steps")

Memory

5 types of agent memory

Memory is what makes an agent useful across sessions. Most developers implement only one type — and wonder why their agent forgets everything. Here's the full picture.

💬
In-context memory
The conversation history in the current session. Temporary — lost when the session ends. Cheap and fast but limited by context window size.
messages = [{"role":"user","content":"..."}]
# grows each turn — watch token cost
📁
External / file memory
Persistent files the agent reads and writes. Survives sessions. Use for: TEAM.md (sprint state), CLAUDE.md (project rules), decision logs, user preferences.
~/.claude/projects/.../memory/
CLAUDE.md, TEAM.md, decisions/log.md
🔍
Semantic memory (RAG)
Embeddings in a vector store. Agent retrieves relevant past context by meaning, not keyword. Use for: large codebases, docs, past decisions, knowledge bases.
ChromaDB (local) / Pinecone (prod)
embed → store → retrieve on query
📋
Procedural memory (skills)
SKILL.md files the agent loads to know HOW to do a task. Skills are your agent's standard operating procedures — loaded on demand, consistent every run.
~/.claude/skills/code-review/SKILL.md
/planning, /tdd, /harness-engineering
📓
Episodic memory (logs)
A record of what the agent did, when, and what happened. Use for: audit trails, debugging agent failures, feeding into future planning. TEAM.md is an example — structured log of daily check-ins.
TEAM.md, decisions/log.md, output/run-2026-05-30.md
# structured entries agent can re-read next session

Multi-agent memory sharing

When multiple agents share state, the key rule is: one writer, many readers. Race conditions corrupt memory files the same way they corrupt databases.

PatternWhen to useRisk to avoid
Shared CLAUDE.mdAll agents read the same project rulesNever let a sub-agent write to it — only the orchestrator
Agent-scoped filesEach agent writes to its own log fileDon't merge at runtime — let the orchestrator aggregate
Central state fileSingle file for shared task state (e.g. sprint.md)Use a file lock or pass state through the orchestrator, not file I/O
Vector store (RAG)Many agents read a shared knowledge baseQueue writes — concurrent embedding jobs corrupt indexes

Memory best practices

Most agent memory problems are not technical — they're about what you choose to store and what you don't.

RuleWhyExample
Store WHY, not WHATThe code shows what exists. Memory needs to explain the decision behind it"Using bcrypt because MD5 was flagged in security audit — do not revert"
Prune stale memoryOld memories mislead agents — they act on things that are no longer trueAfter a refactor, remove architecture memories that describe deleted code
Don't duplicate gitgit log and git blame are authoritative for history — don't replicate them in memory filesUse git log instead of storing "we added caching on May 30"
Keep CLAUDE.md shortIt loads into every session — long files consume tokens on every requestUnder 200 lines. If it grows, split into reference files and load on demand
One file per memory typeMixing user preferences, project state, and references in one file makes pruning harduser.md, feedback-testing.md, project-auth.md, reference-linear.md
CLAUDE.md — correct structure
# Stack (what the agent needs to know about the environment) Python 3.12, FastAPI, PostgreSQL, Redis, Docker # Never do (hard constraints — agent must not violate these) - Never use SQLite, even in tests - Never commit .env files - Never skip tests when adding business logic - Never amend published commits # Always do (defaults the agent should apply) - Read the file before editing it - Use structured outputs (tool_use / JSON) not free text - Write the test before the implementation # Project decisions (the WHY behind current choices) - Auth uses bcrypt: switched from MD5 after security audit Q1 2026 - No ORM: raw SQL for performance, all queries in db/queries/ # References (where to find things) - Sprint tasks: Linear project CORE - Incidents: PagerDuty, escalate to #oncall-eng in Slack

Harness engineering

Why you need an eval harness

AI output is non-deterministic. The same prompt can produce different output next week — because the model changed, the context changed, or you changed the skill. Without a harness, you discover breakage in production.

The silent failure: You update a skill prompt. Your code-review agent now misses SQL injection vulnerabilities because the CRITICAL severity rule shifted from line 40 to line 450 after the edit. Tests never ran. The next PR merges with a security hole. This is not theoretical — it happened.
without a harness
# Monday: skill works perfectly result = run_skill("code-review", code_with_sqli) # → returns CRITICAL: SQL injection # Tuesday: you edit the skill description # Friday: silent regression discovered in prod result = run_skill("code-review", code_with_sqli) # → returns WARNING: consider parameterized queries # → CRITICAL rule is now buried past line 400 — never fires

A harness catches this before merge. It runs the same inputs against the same skill and asserts the output meets defined criteria — every time, in CI.

Harness engineering

3 test types every harness needs

Golden test
Expected output
Given a known input, assert the output contains specific strings, structure, or fields. Catches regressions where the skill stops producing required content.
Adversarial test
Safety rules fire
Give the skill an input that should trigger a safety rule or hard constraint. Assert the rule fires. Catches cases where critical rules get buried and stop executing.
Consistency test
Format doesn't drift
Run the same prompt 3× and compare output structure. If the format changes between runs, the skill lacks a strong enough output template (Pattern 5).
tests/code_review_harness.yaml
golden_tests: - name: detects sql injection input: "SELECT * FROM users WHERE id = " + user_input assert_contains: - "CRITICAL" - "SQL injection" assert_not_contains: - "NIT" # SQL injection must never be downgraded - name: clean code gets approve verdict input: "def add(a: int, b: int) -> int: return a + b" assert_contains: - "APPROVE" assert_not_contains: - "CRITICAL" - "WARNING" adversarial_tests: - name: hardcoded secret triggers critical input: "API_KEY = 'sk-abc123'" assert_contains: - "CRITICAL" - "secret" consistency_tests: - name: severity table always present input: "x = eval(user_input)" runs: 3 assert_all_contain: - "| CRITICAL |" # table must appear every run
Harness engineering

Build the harness

The harness is a Python script that loads your YAML test cases, runs each against the live skill, and asserts the results. It takes about an hour to build the first time.

tests/runner.py
import yaml, anthropic, pathlib, sys from difflib import unified_diff client = anthropic.Anthropic() def load_skill(name: str) -> str: skill_path = pathlib.Path.home() / f".claude/skills/{name}/SKILL.md" return skill_path.read_text() def run_skill(skill_name: str, user_input: str) -> str: skill = load_skill(skill_name) response = client.messages.create( model="claude-sonnet-4-6", max_tokens=2048, system=skill, messages=[{"role": "user", "content": user_input}] ) return response.content[0].text def run_harness(skill_name: str, test_file: str) -> bool: tests = yaml.safe_load(pathlib.Path(test_file).read_text()) failures = [] # Golden tests for t in tests.get("golden_tests", []): result = run_skill(skill_name, t["input"]) for expected in t.get("assert_contains", []): if expected not in result: failures.append(f"FAIL [{t['name']}]: missing '{expected}'") for banned in t.get("assert_not_contains", []): if banned in result: failures.append(f"FAIL [{t['name']}]: found banned '{banned}'") # Adversarial tests (same logic, labeled differently) for t in tests.get("adversarial_tests", []): result = run_skill(skill_name, t["input"]) for expected in t.get("assert_contains", []): if expected not in result: failures.append(f"FAIL [{t['name']}]: safety rule didn't fire — missing '{expected}'") # Consistency tests for t in tests.get("consistency_tests", []): runs = [run_skill(skill_name, t["input"]) for _ in range(t.get("runs", 3))] for required in t.get("assert_all_contain", []): if not all(required in r for r in runs): failures.append(f"FAIL [{t['name']}]: inconsistent — '{required}' missing in some runs") if failures: print("\n".join(failures)) return False print(f"✓ All tests passed for {skill_name}") return True if __name__ == "__main__": ok = run_harness("code-review", "tests/code_review_harness.yaml") sys.exit(0 if ok else 1)

Harness engineering

CI gate — fail before merge, not after

The harness only matters if it runs automatically. Add it to GitHub Actions so every PR that touches a skill or a prompt runs the full test suite before merge is allowed.

.github/workflows/skill-harness.yml
name: Skill Harness on: pull_request: paths: - 'dev-pm-skills/**' # run when any skill changes - 'tests/**' jobs: harness: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Set up Python uses: actions/setup-python@v5 with: python-version: '3.12' - name: Install deps run: pip install anthropic pyyaml - name: Install skills run: ./scripts/install-skills.sh --global - name: Run harness env: ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }} run: python tests/runner.py - name: Post results if: failure() uses: actions/github-script@v7 with: script: | github.rest.issues.createComment({ issue_number: context.issue.number, owner: context.repo.owner, repo: context.repo.repo, body: '⚠️ Skill harness failed. Check the Actions log for which tests broke.' })
Pattern 9 in practice: Eval before merge, not after. This CI job is the technical implementation of that pattern. Every prompt change goes through the harness before it can merge. Regressions become impossible to miss.
Frameworks

Which framework to use

Start with the raw Claude API. Add a framework only when you need what it provides. Most agents that "need LangChain" are just prompt chains with tool calls — which the API handles natively.

Anthropic Claude API
Start here
Native tool use, streaming, vision, and structured outputs. No framework overhead. Full control over the agent loop. Works directly with SKILL.md files as system prompts.
Use when: building a new agent, single-agent workflows, anything under 5 tools. The API handles 80% of real use cases without a framework.
Pydantic AI
Recommended for structured output
Type-safe agent framework. Define your output schema as a Pydantic model — the agent is guaranteed to return valid structured data. Integrates cleanly with FastAPI.
Use when: your agent must return structured data (JSON, typed objects) that downstream code depends on. Pattern 8 (parse structure not text) made easy.
LangGraph
Complex workflows
Graph-based state machine for agents. Each node is a function; edges define transitions. Built for multi-step workflows with branching, looping, and human-in-the-loop checkpoints.
Use when: your agent has complex branching logic, needs checkpointing/resume, or runs long multi-step workflows where you need to inspect and modify state mid-run.
CrewAI
Multi-agent teams
Define agents as roles (Planner, Coder, Reviewer) with skills and goals. Agents collaborate, delegate, and hand off work. Each agent can use a different SKILL.md as its procedure.
Use when: you need multiple specialized agents working in parallel or sequence — e.g., a Planner agent + Implementer agent + Reviewer agent for automated PR review.
LangChain
Use with caution
High-level abstractions over LLM calls, chains, and tool use. Large ecosystem of integrations. But heavy, opinionated, and the abstractions often get in the way as your agent grows.
Use when: you need a specific integration it provides (vector stores, loaders) and can use just that component. Avoid for core agent logic — use the API or LangGraph instead.
Claude Code SDK
For coding agents
Official SDK for building agents that run inside Claude Code. Gives sub-agents access to file tools, bash, and the full skill system. Ideal for CI/CD bots and code generation pipelines.
Use when: building an agent that reads/writes files, runs tests, or operates inside a codebase — not just an API client.
Pydantic AI — structured output example
from pydantic import BaseModel from pydantic_ai import Agent class CodeReview(BaseModel): verdict: str # APPROVE | REQUEST_CHANGES | REJECT critical_count: int issues: list[dict] # [{severity, file, line, message}] summary: str # Load the skill as the agent's system prompt skill = open("~/.claude/skills/code-review/SKILL.md").read() agent = Agent( "claude-sonnet-4-6", system_prompt=skill, result_type=CodeReview # guaranteed structured output ) result = agent.run_sync("Review this PR diff: ...") print(result.data.verdict) # always a valid CodeReview object

Wiring skills into agents

Skills are your agent's procedures. Instead of writing complex prompt logic, load the appropriate SKILL.md as the system prompt for each sub-task. The skill handles the structure; the agent handles the loop.

pr_review_agent.py — skill-powered agent
import anthropic, pathlib client = anthropic.Anthropic() def load_skill(name: str) -> str: return (pathlib.Path.home() / f".claude/skills/{name}/SKILL.md").read_text() def run_with_skill(skill_name: str, task: str) -> str: """Run a task using a specific skill as the system prompt.""" return client.messages.create( model="claude-sonnet-4-6", max_tokens=4096, system=load_skill(skill_name), messages=[{"role": "user", "content": task}] ).content[0].text def review_pr(diff: str, ticket_context: str) -> dict: # Step 1: Security review security = run_with_skill("security", f"Review this diff:\n{diff}") # Step 2: Code quality review quality = run_with_skill("code-review", f"Review this diff:\n{diff}") # Step 3: Write the PR description description = run_with_skill("pr-description", f"Diff:\n{diff}\n\nTicket context:\n{ticket_context}") return { "security": security, "quality": quality, "description": description }

Start with one agent, one skill, one harness

Don't build a multi-agent system on day one. Pick one task your team does manually every day. Wire the right skill to it. Build the harness. Ship it. Then expand. The playbook gives you everything you need: harness-engineering for the tests, planning for the roadmap, code-review for the agent logic itself.

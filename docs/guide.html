Don't vibe code — use agentic coding

Most people use AI wrong: ask, accept, ship. No structure, no tests, no accountability. Here's the difference — and how to do it right.

✕  Vibe coding
✕
Ask AI to write code, accept it, ship it without reading it
✕
No tests — "it worked in dev so it's fine"
✕
Prompt changes silently break behavior you depend on
✕
Output format drifts run to run — automation breaks
✕
No measurable definition of "done" — just vibes
✕
Bugs discovered in production, not before merge
✓  Agentic coding
✓
Use structured skills — every output follows a tested template
✓
Write eval harnesses: golden tests that fail before merge, not after
✓
CI gate catches prompt regressions automatically
✓
Output templates lock the format — automation stays stable
✓
Define "correct" before building — measurable success criteria
✓
Skills + harnesses = AI you can trust in production
Developer path

Your daily coding workflow, upgraded

These skills replace the manual parts of your dev workflow — no copy-pasting prompts, no extra setup per project.

1

Install once — works in every project

Clone the repo and run the install script. Skills go into ~/.claude/skills/ globally — no setup per project.

terminal
git clone https://github.com/majidraza1228/claude-skills.git cd claude-skills ./scripts/install-skills.sh --global
2

Open Claude Code and start typing

No slash commands needed. Just describe what you're doing — the right skill loads automatically.

"review my code" → severity table + verdict "why is this failing" → 5-step debug triage "write a commit message" → conventional commit "write tests for this" → red-green-refactor TDD

5

Don't ship AI output without a harness

This is the step most developers skip — and where vibe coding starts. Every AI-generated function, query, or review should have a test that confirms it behaves correctly. Use the harness-engineering skill to build one.

Claude Code
"build an eval harness for my code-review skill" → YAML golden test cases + Python assertion runner → GitHub Actions CI gate that fails before merge
🧪

What the harness catches

A golden test runs the same prompt and asserts the output matches expected structure. An adversarial test checks that safety rules fire. A consistency test runs the same prompt 3× and checks format doesn't drift. These run in CI on every PR — so prompt regressions never reach production. See the harness-engineering skill →

4

Define success before you trigger any skill

This is where PM vibe-coding happens: asking AI to plan a sprint or write a PRD without knowing what "done" looks like. The output sounds good but has no teeth. Fix this with one rule: write your success criteria before you trigger the skill.

Before you triggerAsk yourself
ticket-writerWhat does acceptance look like? What are the edge cases?
sprint-planningWhat does a successful sprint deliver — specifically?
prd-writerHow will you measure if this feature worked in 30 days?
retroWhat one action item must come out of this, with a named owner?
📏

Pattern 10: Define "correct" before building

If you can't write a test that proves the feature worked, you're not ready to build it. The same applies to AI output — if you can't define what a good ticket looks like, the AI can't produce one reliably. See Pattern 10 →

5

Set up the agent PM system (optional, but powerful)

Developers check in once a day. You get a full team summary on demand — no chasing updates.

terminal — run once in your project
# Copy templates to your project root cp templates/TEAM.md ./TEAM.md cp templates/SPRINT.md ./SPRINT.md # Install the two agent PM skills ./scripts/install-skills.sh --project daily-checkin team-digest

Each dev runs /daily-checkin in 30 seconds. You run /team-digest any time to see sprint velocity, blockers, and who's stuck.

Tip: Fill SPRINT.md with your tasks and owners before the sprint starts. The system reads it automatically — no extra setup during the sprint.

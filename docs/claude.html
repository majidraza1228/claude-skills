Foundations

What is agentic coding?

A single prompt is a request. An agent is a loop. It plans, acts, observes the result, and decides what to do next — autonomously, using tools and memory.

Vibe coding treats every AI interaction as a one-shot prompt. You ask, it answers, you move on. There's no loop, no verification, no accountability. This breaks at scale.
ApproachHow it worksFailure mode
PromptAsk once, accept the answerNo verification — bugs ship
SkillStructured procedure, consistent outputNo loop — one task at a time
AgentPlan → act → verify → loopNeeds harness or it drifts
Skill + AgentAgent runs reliable sub-proceduresCorrect approach ✓

agent_loop.py — minimal pattern
# Goal → Plan → Act → Observe → Verify → Loop import anthropic client = anthropic.Anthropic() def run_agent(goal: str, tools: list, max_steps: int = 10): messages = [{"role": "user", "content": goal}] for step in range(max_steps): response = client.messages.create( model="claude-sonnet-4-6", max_tokens=4096, tools=tools, messages=messages ) # Agent is done — no more tool calls if response.stop_reason == "end_turn": return response.content[0].text # Execute tool calls and feed results back tool_results = execute_tools(response.content) messages.append({"role": "assistant", "content": response.content}) messages.append({"role": "user", "content": tool_results}) raise RuntimeError(f"Agent exceeded {max_steps} steps")

Memory

5 types of agent memory

Memory is what makes an agent useful across sessions. Most developers implement only one type — and wonder why their agent forgets everything. Here's the full picture.

💬
In-context memory
The conversation history in the current session. Temporary — lost when the session ends. Cheap and fast but limited by context window size.
messages = [{"role":"user","content":"..."}]
# grows each turn — watch token cost
📁
External / file memory
Persistent files the agent reads and writes. Survives sessions. Use for: TEAM.md (sprint state), CLAUDE.md (project rules), decision logs, user preferences.
~/.claude/projects/.../memory/
CLAUDE.md, TEAM.md, decisions/log.md
🔍
Semantic memory (RAG)
Embeddings in a vector store. Agent retrieves relevant past context by meaning, not keyword. Use for: large codebases, docs, past decisions, knowledge bases.
ChromaDB (local) / Pinecone (prod)
embed → store → retrieve on query
📋
Procedural memory (skills)
SKILL.md files the agent loads to know HOW to do a task. Skills are your agent's standard operating procedures — loaded on demand, consistent every run.
~/.claude/skills/code-review/SKILL.md
/planning, /tdd, /harness-engineering
📓
Episodic memory (logs)
A record of what the agent did, when, and what happened. Use for: audit trails, debugging agent failures, feeding into future planning. TEAM.md is an example — structured log of daily check-ins.
TEAM.md, decisions/log.md, output/run-2026-05-30.md
# structured entries agent can re-read next session

Multi-agent memory sharing

When multiple agents share state, the key rule is: one writer, many readers. Race conditions corrupt memory files the same way they corrupt databases.

PatternWhen to useRisk to avoid
Shared CLAUDE.mdAll agents read the same project rulesNever let a sub-agent write to it — only the orchestrator
Agent-scoped filesEach agent writes to its own log fileDon't merge at runtime — let the orchestrator aggregate
Central state fileSingle file for shared task state (e.g. sprint.md)Use a file lock or pass state through the orchestrator, not file I/O
Vector store (RAG)Many agents read a shared knowledge baseQueue writes — concurrent embedding jobs corrupt indexes

Memory best practices

Most agent memory problems are not technical — they're about what you choose to store and what you don't.

RuleWhyExample
Store WHY, not WHATThe code shows what exists. Memory needs to explain the decision behind it"Using bcrypt because MD5 was flagged in security audit — do not revert"
Prune stale memoryOld memories mislead agents — they act on things that are no longer trueAfter a refactor, remove architecture memories that describe deleted code
Don't duplicate gitgit log and git blame are authoritative for history — don't replicate them in memory filesUse git log instead of storing "we added caching on May 30"

Harness engineering

Why you need an eval harness

AI output is non-deterministic. The same prompt can produce different output next week — because the model changed, the context changed, or you changed the skill. Without a harness, you discover breakage in production.

The silent failure: You update a skill prompt. Your code-review agent now misses SQL injection vulnerabilities because the CRITICAL severity rule shifted from line 40 to line 450 after the edit. Tests never ran. The next PR merges with a security hole. This is not theoretical — it happened.
without a harness
# Monday: skill works perfectly result = run_skill("code-review", code_with_sqli) # → returns CRITICAL: SQL injection # Tuesday: you edit the skill description # Friday: silent regression discovered in prod result = run_skill("code-review", code_with_sqli) # → returns WARNING: consider parameterized queries # → CRITICAL rule is now buried past line 400 — never fires

A harness catches this before merge. It runs the same inputs against the same skill and asserts the output meets defined criteria — every time, in CI.

Harness engineering

3 test types every harness needs

Golden test
Expected output
Given a known input, assert the output contains specific strings, structure, or fields. Catches regressions where the skill stops producing required content.
Adversarial test
Safety rules fire
Give the skill an input that should trigger a safety rule or hard constraint. Assert the rule fires. Catches cases where critical rules get buried and stop executing.
Consistency test
Format doesn't drift
Run the same prompt 3× and compare output structure. If the format changes between runs, the skill lacks a strong enough output template (Pattern 5).
tests/code_review_harness.yaml
golden_tests: - name: detects sql injection input: "SELECT * FROM users WHERE id = " + user_input assert_contains: - "CRITICAL" - "SQL injection" assert_not_contains: - "NIT" # SQL injection must never be downgraded - name: clean code gets approve verdict input: "def add(a: int, b: int) -> int: return a + b" assert_contains: - "APPROVE" assert_not_contains: - "CRITICAL" - "WARNING" adversarial_tests: - name: hardcoded secret triggers critical input: "API_KEY = 'sk-abc123'" assert_contains: - "CRITICAL" - "secret" consistency_tests: - name: severity table always present input: "x = eval(user_input)" runs: 3 assert_all_contain: - "| CRITICAL |" # table must appear every run
Harness engineering

Build the harness

The harness is a Python script that loads your YAML test cases, runs each against the live skill, and asserts the results. It takes about an hour to build the first time.

tests/runner.py
import yaml, anthropic, pathlib, sys from difflib import unified_diff client = anthropic.Anthropic() def load_skill(name: str) -> str: skill_path = pathlib.Path.home() / f".claude/skills/{name}/SKILL.md" return skill_path.read_text() def run_skill(skill_name: str, user_input: str) -> str: skill = load_skill(skill_name) response = client.messages.create( model="claude-sonnet-4-6", max_tokens=2048, system=skill, messages=[{"role": "user", "content": user_input}] ) return response.content[0].text def run_harness(skill_name: str, test_file: str) -> bool: tests = yaml.safe_load(pathlib.Path(test_file).read_text()) failures = [] # Golden tests for t in tests.get("golden_tests", []): result = run_skill(skill_name, t["input"]) for expected in t.get("assert_contains", []): if expected not in result: failures.append(f"FAIL [{t['name']}]: missing '{expected}'") for banned in t.get("assert_not_contains", []): if banned in result: failures.append(f"FAIL [{t['name']}]: found banned '{banned}'") # Adversarial tests (same logic, labeled differently) for t in tests.get("adversarial_tests", []): result = run_skill(skill_name, t["input"]) for expected in t.get("assert_contains", []): if expected not in result: failures.append(f"FAIL [{t['name']}]: safety rule didn't fire — missing '{expected}'") # Consistency tests for t in tests.get("consistency_tests", []): runs = [run_skill(skill_name, t["input"]) for _ in range(t.get("runs", 3))] for required in t.get("assert_all_contain", []): if not all(required in r for r in runs): failures.append(f"FAIL [{t['name']}]: inconsistent — '{required}' missing in some runs") if failures: print("\n".join(failures)) return False print(f"✓ All tests passed for {skill_name}") return True if __name__ == "__main__": ok = run_harness("code-review", "tests/code_review_harness.yaml") sys.exit(0 if ok else 1)

CI gate — fail before merge, not after

The harness only matters if it runs automatically. Add it to GitHub Actions so every PR that touches a skill or a prompt runs the full test suite before merge is allowed.

.github/workflows/skill-harness.yml
name: Skill Harness on: pull_request: paths: - 'dev-pm-skills/**' # run when any skill changes - 'tests/**' jobs: harness: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Set up Python uses: actions/setup-python@v5 with: python-version: '3.12' - name: Install deps run: pip install anthropic pyyaml - name: Install skills run: ./scripts/install-skills.sh --global - name: Run harness env: ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }} run: python tests/runner.py - name: Post results if: failure() uses: actions/github-script@v7 with: script: | github.rest.issues.createComment({ issue_number: context.issue.number, owner: context.repo.owner, repo: context.repo.repo, body: '⚠️ Skill harness failed. Check the Actions log for which tests broke.' })
Pattern 9 in practice: Eval before merge, not after. This CI job is the technical implementation of that pattern. Every prompt change goes through the harness before it can merge. Regressions become impossible to miss.
Frameworks

Which framework to use

Start with the raw Claude API. Add a framework only when you need what it provides. Most agents that "need LangChain" are just prompt chains with tool calls — which the API handles natively.

Anthropic Claude API
Start here
Native tool use, streaming, vision, and structured outputs. No framework overhead. Full control over the agent loop. Works directly with SKILL.md files as system prompts.
Use when: building a new agent, single-agent workflows, anything under 5 tools. The API handles 80% of real use cases without a framework.
Pydantic AI
Recommended for structured output
Type-safe agent framework. Define your output schema as a Pydantic model — the agent is guaranteed to return valid structured data. Integrates cleanly with FastAPI.
Use when: your agent must return structured data (JSON, typed objects) that downstream code depends on. Pattern 8 (parse structure not text) made easy.
LangGraph
Complex workflows
Graph-based state machine for agents. Each node is a function; edges define transitions. Built for multi-step workflows with branching, looping, and human-in-the-loop checkpoints.
Use when: your agent has complex branching logic, needs checkpointing/resume, or runs long multi-step workflows where you need to inspect and modify state mid-run.
CrewAI
Multi-agent teams
Define agents as roles (Planner, Coder, Reviewer) with skills and goals. Agents collaborate, delegate, and hand off work. Each agent can use a different SKILL.md as its procedure.
Use when: you need multiple specialized agents working in parallel or sequence — e.g., a Planner agent + Implementer agent + Reviewer agent for automated PR review.
LangChain
Use with caution
High-level abstractions over LLM calls, chains, and tool use. Large ecosystem of integrations. But heavy, opinionated, and the abstractions often get in the way as your agent grows.
Use when: you need a specific integration it provides (vector stores, loaders) and can use just that component. Avoid for core agent logic — use the API or LangGraph instead.
Claude Code SDK
For coding agents
Official SDK for building agents that run inside Claude Code. Gives sub-agents access to file tools, bash, and the full skill system. Ideal for CI/CD bots and code generation pipelines.
Use when: building an agent that reads/writes files, runs tests, or operates inside a codebase — not just an API client.

Sub-agents

What are sub-agents?

A sub-agent is a Claude instance spawned by another Claude instance (the orchestrator) to handle a specific sub-task. The orchestrator plans and delegates; sub-agents execute and report back. Each runs in its own context window — isolated, focused, and replaceable.

Orchestrator
→
Sub-agent: Security
Sub-agent: Tests
Sub-agent: Docs
→
Merged result
ProsConsWhen it's worth it
Parallel execution — multiple sub-agents run simultaneously, cutting wall-clock time Token cost multiplies — each sub-agent loads its own context, so you pay per agent Tasks with independent parts that don't need to share state mid-run
Specialization — each sub-agent gets its own system prompt (skill), making it better at one job No shared context — sub-agents can't read each other's messages; orchestrator must relay information explicitly Different sub-tasks benefit from different expertise (security vs. docs vs. tests)
Context isolation — a large codebase won't overflow the orchestrator's window; sub-agents handle the deep reads Coordination overhead — orchestrator must parse, merge, and validate each sub-agent's output Sub-tasks each need to read large amounts of context that would overwhelm a single agent
Failure isolation — a crashed sub-agent doesn't kill the whole workflow; retry just that worker Harder to debug — errors can occur in any sub-agent; you need logging at the orchestrator level Reliability matters and individual steps can be retried independently
Don't default to sub-agents. A single well-prompted agent with the right tools handles most tasks better and cheaper. Use sub-agents when you have genuinely parallel work, context window pressure, or tasks that need different specialist prompts.
Sub-agents

The orchestrator pattern

The orchestrator breaks a goal into tasks, assigns each to a sub-agent, collects the results, and merges them into a final output. Sub-agents are stateless workers — they get a task, they return a result.

orchestrator.py — parallel sub-agents
import anthropic, asyncio, pathlib client = anthropic.Anthropic() def load_skill(name: str) -> str: return (pathlib.Path.home() / f".claude/skills/{name}/SKILL.md").read_text() def run_subagent(skill_name: str, task: str) -> str: """A focused worker: one skill, one task, one result.""" return client.messages.create( model="claude-sonnet-4-6", max_tokens=4096, system=load_skill(skill_name), messages=[{"role": "user", "content": task}] ).content[0].text def orchestrate_pr_review(pr_diff: str) -> dict: """Orchestrator: run three specialist sub-agents in parallel.""" import concurrent.futures tasks = { "security": ("code-review", f"Security audit only:\n{pr_diff}"), "tests": ("tdd", f"Identify missing tests:\n{pr_diff}"), "docs": ("doc-writer", f"Flag undocumented public API changes:\n{pr_diff}"), } results = {} with concurrent.futures.ThreadPoolExecutor() as pool: futures = { pool.submit(run_subagent, skill, prompt): name for name, (skill, prompt) in tasks.items() } for future in concurrent.futures.as_completed(futures): results[futures[future]] = future.result() # Orchestrator merges sub-agent outputs into a final review merged = client.messages.create( model="claude-sonnet-4-6", max_tokens=2048, messages=[{"role": "user", "content": f"Merge these specialist reviews into one prioritised list:\n\n" f"SECURITY:\n{results['security']}\n\n" f"TESTS:\n{results['tests']}\n\n" f"DOCS:\n{results['docs']}" }] ).content[0].text return {"sub_agent_results": results, "merged_review": merged}
RuleWhy
Stateless workers — sub-agents take input, return output, nothing moreMakes them retryable and swappable without side effects
Orchestrator owns state — all shared data flows through the orchestrator, never file I/O between sub-agentsPrevents race conditions and data corruption
One skill per sub-agent — load a single SKILL.md as the system promptKeeps the sub-agent focused; generalist prompts produce generalist output
Validate before merging — check sub-agent output structure before passing to orchestrator merge stepA malformed sub-agent response shouldn't corrupt the final output
Log each sub-agent call — record input, output, latency, and model versionDebugging a multi-agent pipeline is impossible without per-agent logs
Sub-agents

Claude Code workflows

Claude Code has a built-in Agent tool that lets it spawn sub-agents natively — without any Python orchestrator code. You describe the workflow in a prompt or skill; Claude handles the delegation. The Claude Code SDK lets you go further and drive multi-agent pipelines programmatically.

How it works in Claude Code

When you give Claude Code a complex task, it can spawn a sub-agent (another claude process) to handle part of it — in a git worktree so the sub-agent works in isolation. You don't write orchestration code; Claude decides when delegation makes sense.

tell Claude Code to use sub-agents
# Natural language — Claude decides when to spawn "Review this PR in parallel: run security checks, generate missing tests, and update the changelog. Use sub-agents so the three tasks run at the same time." # Or wire it into a skill's instructions (SKILL.md) # The skill tells Claude how to delegate: "Spawn three sub-agents: one for /code-review (security focus), one for /tdd (test gaps), one for /doc-writer (API changes). Merge their outputs into a single prioritised comment."

Claude Code SDK — programmatic workflows

The Claude Code SDK lets you drive Claude Code from Python or TypeScript — useful for CI pipelines, scheduled workflows, and production agents that need the full Claude Code toolset (bash, file editing, git) without a human in the loop.

workflow.py — Claude Code SDK
import asyncio from claude_code_sdk import query, ClaudeCodeOptions async def run_pr_workflow(pr_diff: str): # Each query() call is a Claude Code session with full tool access # (bash, file read/write, git) — not just raw API options = ClaudeCodeOptions( max_turns=10, system_prompt="You are a senior engineer. Be concise and precise." ) # Run specialist sub-agents in parallel using asyncio security_task = query( prompt=f"Security review this diff. Flag CRITICAL issues only:\n{pr_diff}", options=options ) tests_task = query( prompt=f"List missing test cases for this diff:\n{pr_diff}", options=options ) security_result, tests_result = await asyncio.gather(security_task, tests_task) # Collect text from streaming message events security_text = "".join( chunk.text for chunk in security_result if hasattr(chunk, "text") ) tests_text = "".join( chunk.text for chunk in tests_result if hasattr(chunk, "text") ) return {"security": security_text, "tests": tests_text} asyncio.run(run_pr_workflow(open("pr.diff").read()))

Worktrees — isolated sub-agent environments

When Claude Code spawns a sub-agent for file-editing tasks, it creates a git worktree — a separate checkout of the repo the sub-agent can modify without touching your working branch. Changes merge back (or get discarded) when the sub-agent is done.

Don't start with five sub-agents. Start with one orchestrator and one worker: "Use a sub-agent to run /code-review on this PR diff. Then summarise the findings for me." Once that works reliably, add a second worker. Complexity compounds — add agents one at a time and test each addition.

Workflow scripts

What is a Claude workflow?

A workflow is a workflow.js script that coordinates multiple LLM calls, tools, and sub-agents to accomplish a complex goal. Unlike a chat window — where everything accumulates in one context — workflows use isolated sub-agents to prevent context bloat and ensure deterministic, repeatable execution.

ConceptWhat it does
workflow.jsThe script that defines phases, agent assignments, fan-out patterns, and merge steps. Runs as a separate process — not inside the chat window.
JournalPersists the state of a running workflow. Lets you pause mid-run and resume later without losing completed agent results.
Isolated sub-agentsEach sub-agent gets its own context window. No shared state between workers — only the orchestrator sees all outputs. Prevents context bloat.
Fan-outOne orchestrator spawns many parallel workers at the same time. Results collected once all workers complete.
Adversarial patternMultiple agents independently evaluate the same output, then a final agent synthesises the consensus. Catches errors a single agent would miss.

Chat vs Workflow: In a chat window, every message adds to one growing context. After 20 tool calls the window fills up, the model loses earlier context, and quality degrades. A workflow runs each phase in a fresh sub-agent — context stays clean at every step regardless of total workflow size.

Uses different Claude models for different phases — cheapest model for volume work, best model only for the final output. Dramatically cuts token cost without sacrificing quality.

// Phase 1 — Generate: 6 Haiku agents brainstorm brand names + taglines // Haiku = fastest + cheapest. Workflow scripts

Build your own workflow

A workflow script has three responsibilities: define phases, assign agents (with model + skill), and handle the Journal so runs can pause and resume. Start minimal — one fan-out, one merge.

workflow.js — minimal structure
import { workflow, fanOut, journal } from "@anthropic-ai/claude-code/workflow"; export default workflow({ name: "pr-review", journal: journal("./.claude/journal/pr-review.json"), // pause + resume phases: [ { name: "analyse", // Fan out: 3 parallel specialist sub-agents run: fanOut([ { skill: "code-review", model: "claude-sonnet-4-6", input: "{{pr_diff}}" }, { skill: "tdd", model: "claude-haiku-4-5", input: "{{pr_diff}}" }, { skill: "doc-writer", model: "claude-haiku-4-5", input: "{{pr_diff}}" }, ]), }, { name: "merge", // Single Opus call to synthesise all findings run: { model: "claude-opus-4-7", input: "{{analyse.results}}" }, }, ], });

MCP

Model Context Protocol

MCP is the standard protocol that lets Claude connect to external tools, data sources, and services — without embedding integration code in your app. You run an MCP server; Claude calls it like a tool.

Why it matters for agents: Without MCP, every agent integration (GitHub, Slack, your database, your filesystem) requires custom tool definitions in your code. With MCP, each integration is a standalone server. Swap, add, or version servers without touching the agent loop.
ApproachHow it worksBest for
Native tool_useTool definitions live in your Python/JS code alongside the API callSimple, project-specific tools that won't be reused
MCP serverSeparate process exposes tools over a standard protocol; any Claude client connectsReusable integrations: GitHub, databases, file systems, internal APIs
Claude Code built-inClaude Code ships with filesystem, bash, browser tools out of the boxDay-to-day development — no setup needed

Configure MCP servers

MCP servers are configured in a JSON file. Claude Code reads this on startup and makes all listed servers available as tools automatically.

~/.claude/claude_desktop_config.json — global config
{ "mcpServers": { // Local filesystem access (built-in, usually auto-configured) "filesystem": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/you/projects"] }, // GitHub — needs a personal access token "github": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"], "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_..." } }, // Your own internal API server (see next section) "my-api": { "command": "python", "args": ["/path/to/mcp_server.py"] } } }

MCP

Build a custom MCP server

If you have an internal API, CLI tool, or data source you want Claude to use, wrap it as an MCP server. The Python SDK makes this ~30 lines of code.

mcp_server.py — minimal custom server
from mcp.server.fastmcp import FastMCP import httpx mcp = FastMCP("my-internal-api") # Each @mcp.tool() becomes a callable tool inside Claude @mcp.tool() def get_sprint_tasks(team: str) -> list[dict]: """Fetch active sprint tasks for a team from the internal tracker.""" resp = httpx.get(f"https://tracker.internal/api/sprint/{team}") return resp.json()["tasks"] @mcp.tool() def create_ticket(title: str, description: str, priority: str = "medium") -> dict: """Create a new ticket in the internal tracker.""" resp = httpx.post("https://tracker.internal/api/tickets", json={"title": title, "description": description, "priority": priority}) return resp.json() @mcp.resource("docs://{page}") def get_doc_page(page: str) -> str: """Serve internal documentation pages as MCP resources.""" return (Path("docs") / f"{page}.md").read_text() if __name__ == "__main__": mcp.run() # stdio transport — Claude Code connects automatically

MCP in an agent loop

When you register an MCP server, its tools appear alongside native tool_use in the agent response. The agent calls them the same way — no extra code required.

agent_with_mcp.py
import anthropic client = anthropic.Anthropic() # No tool definitions needed — MCP servers inject their tools automatically # when using Claude Code. For the raw API, use the MCP client SDK: from mcp import ClientSession, StdioServerParameters from mcp.client.stdio import stdio_client async def run_agent_with_mcp(goal: str): server_params = StdioServerParameters( command="python", args=["mcp_server.py"] ) async with stdio_client(server_params) as (read, write): async with ClientSession(read, write) as session: await session.initialize() # List available tools from the MCP server tools = await session.list_tools() tool_defs = [t.model_dump() for t in tools.tools] response = client.messages.create( model="claude-sonnet-4-6", max_tokens=4096, tools=tool_defs, messages=[{"role": "user", "content": goal}] ) return response

Wiring skills into agents

Skills are your agent's procedures. Instead of writing complex prompt logic, load the appropriate SKILL.md as the system prompt for each sub-task. The skill handles the structure; the agent handles the loop.

pr_review_agent.py — skill-powered agent
import anthropic, pathlib client = anthropic.Anthropic() def load_skill(name: str) -> str: return (pathlib.Path.home() / f".claude/skills/{name}/SKILL.md").read_text() def run_with_skill(skill_name: str, task: str) -> str: """Run a task using a specific skill as the system prompt.""" return client.messages.create( model="claude-sonnet-4-6", max_tokens=4096, system=load_skill(skill_name), messages=[{"role": "user", "content": task}] ).content[0].text def review_pr(diff: str, ticket_context: str) -> dict: # Step 1: Security review security = run_with_skill("security", f"Review this diff:\n{diff}") # Step 2: Code quality review quality = run_with_skill("code-review", f"Review this diff:\n{diff}") # Step 3: Write the PR description description = run_with_skill("pr-description", f"Diff:\n{diff}\n\nTicket context:\n{ticket_context}") return { "security": security, "quality": quality, "description": description }

11 patterns

Patterns that actually work

Built from 75+ tests. Every pattern fixes a specific failure mode — skills that don't trigger, output that drifts, AI features shipped without success criteria.

⚡
Skill Design — for anyone writing SKILL.md files
Patterns 1–7. Each one fixes a failure mode found across 75+ real skill tests.
01
Descriptions ≥ 150 chars
Skill never triggers

Claude matches user prompts against skill descriptions at routing time. A 54-character description has too few words to match on — the skill stays invisible.

Before — 54 chars
description: "Review code for bugs and style issues"
After — 280 chars
description: > Use this skill when the user asks for a code review, wants feedback on their code, asks to find bugs, requests style checks, or pastes code and says 'what do you think' or 'anything wrong here'. Covers Python, JavaScript, TypeScript, Go, Rust.
Rule: Target 150–300 chars. Add 3–5 alternate trigger phrases users actually say. Include languages or domains the skill covers.
02
Exclusions belong in the description
Wrong skill loads

"Do not use for spreadsheets" on line 45 of the skill body fires after the wrong skill has already loaded. Descriptions are read at routing time; the body is read at execution time.

Before — exclusion in body (too late)
# line 45 of SKILL.md body: Do not use this skill for spreadsheets.
After — exclusion in description
description: > Use for Word documents (.docx). Do NOT use for spreadsheets — use /xlsx. Do NOT use for PDFs — use /pdf.
Rule: Every "do not use for X" must include "use /Y instead." Without a redirect, Claude knows this skill is wrong but doesn't know what's right.
03
Direct imperative tone
Vague, hedging output

Claude mirrors the tone of instructions. Hedging language produces hedging output.

Before — tentative
Could you take a look and maybe check if there are any issues?
Output: "You might want to consider renaming this variable — it could be a bit clearer."
After — direct
Flag every issue. Classify severity: critical, warning, nit. Reference exact line. Do not soften language.
Output: "| CRITICAL | auth.py:42 | SQL injection |"
Rule: Replace hedge words (maybe, could, might, consider) with imperatives: Flag, List, Output, Return, Classify.
04
Three-column source tables
Missing or wrong data

"Check the relevant files" forces Claude to guess which files, where they are, and what to extract. It guesses wrong.

Before — vague
Before generating the report, check the relevant configuration files and extract the necessary settings.
After — explicit table
| Source | Path | Extract | |-------------|-------------------|----------------| | App config | /config/app.yaml | name, version | | DB settings | /config/db.yml | host, port | | Flags | /config/flags.json| enabled keys |
Rule: Replace every "check", "look at", or "review" without specifics with a Source / Path / What to Extract table.
05
Output templates prevent drift
Format changes every run

Same skill, same prompt, three mornings, three different output structures. "Analyze the git commit and write a summary" → Monday: plain text. Tuesday: markdown headers. Wednesday: bullet list.

Output template
Format every commit analysis using this exact template: --- **Commit:** {short_hash} — {title} **Risk:** {Critical | High | Medium | Low} **What changed:** - {file_path}: {one-line description} **Why it matters:** {1-2 sentences on impact} ---
Rule: Define exact output structure. Use {placeholder} syntax. Run the same prompt 3× — if format differs, add a template.
06
One example beats five rules
Inconsistent structure

12 formatting rules → inconsistent output across runs. 2 worked examples → identical output every time. Rules require interpretation. An example showing fix: Add retry logic is unambiguous.

Before — rules only
1. Use imperative mood 2. Start with type prefix 3. Subject under 72 chars 4. Blank line after subject 5. Explain why, not how ...8 more rules...
After — worked example
INPUT: Added retry logic for payment timeouts. Closes #234. OUTPUT: fix: Add retry logic for payment timeouts Add try/catch with exponential backoff. Retry up to 3 times before failing. Closes #234
Rule: For every skill with more than 5 formatting rules, add 1–2 worked examples with real input → exact expected output.
07
Skills under 500 lines
Bottom-half instructions ignored

In an 800-line fitness skill, safety rules at line 700+ never fired. Not once in 75+ tests. Claude's attention to instructions degrades with length — line position is a priority ranking.

Attention map
Lines 1–100: Core logic ✓ Always works Lines 100–200: Feature detail ✓ Always works Lines 200–350: Secondary rules ⚠ Usually works Lines 350–500: Edge cases ⚠ Sometimes Lines 500–650: Recovery logic ✗ Unreliable Lines 650–800: Safety warnings ✗ Never fires
Fix A: Move safety-critical rules to the first 100 lines. Fix B: Split skills over 500 lines into focused sub-skills — see dev-pm-skills/fitness-coach/ for the pattern.

Building with AI — for developers and PMs
Patterns 8–11. For teams shipping AI features, not just writing skills.
08
Parse structure, not text
Regex on AI output breaks silently

Free-text output looks stable until it isn't. One preamble sentence — "Here's my review:" — and response.split('\n')[0] returns the wrong value with no error. Use tool use / JSON schema for any AI output consumed by code.

Before — fragile
text = response.content[0].text severity = text.split('\n')[0] .split(':')[1].strip() # Breaks if model adds preamble
After — tool use
tools = [{"name": "review_result", "input_schema": { "properties": { "severity": {"enum": ["critical",...]}, "issues": {"type": "array"} }}}] result = response.content[0].input # Always a dict — no parsing needed
Rule: Any AI output consumed by code must use tool use or a JSON schema. Free-text output is fine for humans — not for anything downstream.
09
Eval before merge, not after
Prompt changes silently break behavior

Developers run tests before merging code. They don't run tests before merging a SKILL.md change. A prompt is treated like documentation, not like code — until it breaks production.

.github/workflows/eval.yml
name: Skill evals on: [push, pull_request] jobs: eval: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: pip install anthropic pyyaml - run: python evals/runner.py evals/code-review.yaml dev-pm-skills/code-review/SKILL.md env: ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }} # Exits 1 on any failure — blocks merge
Rule: A SKILL.md change without an eval run is a deploy without tests. See harness-engineering skill for the full runner setup.
10
Define "correct" before building
No way to know if the feature is working

PMs write acceptance criteria like "the AI should give relevant suggestions." Relevant is not a test. The AI sounds equally confident whether right or wrong — without measurable criteria, there's no way to ship with confidence.

Before — unmeasurable
Acceptance criteria: - AI generates 3+ suggestions - Suggestions are relevant - Response loads in <5s
After — measurable
Correctness: 8/10 match expert golden set Precision: 0 contradict a dependency Latency: P95 < 3s at launch Kill threshold: acceptance rate < 20% = rollback or rework
Rule: Write measurable success criteria before the first prompt. Set a kill threshold — if metric Y drops below Z, the feature gets rolled back.
11
Token cost is a product constraint
$0.40/request discovered at billing

A feature costing $0.40/request looks fine with 10 test users. At 10,000 DAUs it's $4,000/day. Devs think in latency. PMs think in user value. Neither thinks in token budget during design.

Before — unbounded context
client.messages.create( model="claude-opus-4-6", # always Opus system=full_prompt, # 2k tokens messages=full_history # grows forever ) # Day 1: 3k tokens. Day 30: 40k tokens.
After — token-aware
# Cache static system prompt (once) # Summarise history > 5 turns # Chunk docs — relevant section only # Use Haiku for triage, Sonnet for gen model = "claude-haiku-4-5" # 10× cheaper
Rule: Estimate token cost per request at design time: (system + avg context + avg response) × model price. Pick model tier by task — don't default to Opus for everything.
